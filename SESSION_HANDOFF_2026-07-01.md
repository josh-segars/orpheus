## What this session did

### The upload gate trio (ORPHEUS-88 + ORPHEUS-100 + ORPHEUS-101) — commit `3a73bfc`, migration 016

POST /jobs now runs **three ordered gates** so a bad upload never becomes an authoritative-looking report. All three surface a 422 through GroundworkPage's existing `ApiError.body.detail` error slot (no frontend change for the reject messages).

- **2a — filename type (ORPHEUS-101).** `parse_archive_filename()` in `zip_parser.py` reads the real `Complete_LinkedInDataExport_MM-DD-YYYY.zip` shape (case-insensitive prefix, basename from any path, MM-DD-YYYY date, invalid date → None; both fields independently optional). A `Basic_` prefix rejects immediately with the Complete-archive guidance (shared `_BASIC_ARCHIVE_DETAIL` constant).
- **2b — content missing-file (ORPHEUS-88).** Backstop for a *renamed* Basic/corrupt archive: blocks only `CRITICAL`+`MISSING_FILE` (Profile.csv/Shares.csv absent). An `EMPTY_DATA` critical (zero behavioral on an otherwise-Complete archive) is a **valid low-signal report** for a genuinely inactive member and passes through — this was the key design call ([Josh, 2026-07-01], product application touching framework territory).
- **2c — recency (ORPHEUS-100 + ORPHEUS-101).** Reject when the export date is >14 days old. Export date = the filename's MM-DD-YYYY stamp when present, else the analytics XLSX's newest dated row (`latest_analytics_date()` in `xlsx_parser.py` — max of ENGAGEMENT/FOLLOWERS daily rows, fallback the DISCOVERY period-end). The XLSX is the freshness signal because "Past 365 days" analytics refresh continuously; the ZIP's activity dates would false-positive on inactive members (the ORPHEUS-91 trap).

**Data-limited surface.** Reports that complete on degraded data (allowed-through EMPTY_DATA critical + data-limitation warnings) carry a denormalized `jobs.data_limited` boolean (migration 016), set by the worker at completion. It drives a client **report banner** (full notices via GET /jobs/{id}'s new `quality` payload block) and a **"Limited data" chip** on the reports list, advisor roster, and `/admin`. New `DataQualityReport` classifier helpers (`blocking_issues`/`has_blocking_issue`/`data_limitation_issues`/`is_data_limited`/`data_limitation_notices`) separate blocking (CRITICAL+MISSING_FILE) from data-limitation (CRITICAL/WARNING in {missing_file, empty_data, parse_failure, date_range}; MISSING_FIELD + INFO excluded).

**No backfill (deliberate).** A probe found 28 stored jobs with a critical/missing_file flag, but those were generated by the pre-ORPHEUS-87 parser blind to suffixed CSVs (over-reports missing files). Backfilling would have mislabeled ~33 healthy reports; the backfill was run and reverted (all 38 rows back to false). The flag populates only for jobs scored going forward — same posture as ORPHEUS-97.

**Design decisions locked** (all [Josh, 2026-07-01], all product application / ingestion policy): hard-block Basic + stale (not warn); 14-day staleness threshold; EMPTY_DATA passes through as valid; filename-first with content backstop; client banner + advisor/admin chip. No new cross-stakeholder Decision Log entry needed.
